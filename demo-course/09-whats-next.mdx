The eight scenarios were a self-contained tour of what Computer Use can do today, ending with the moment a user packages a recurring task into a skill. This page is about what happens after that.

## The mental model

A Yambr skill is just a folder with code and a `SKILL.md` describing it. Three properties make that folder useful beyond the chat:

1. **It has a stable name.** `invoice-builder` is the same thing tomorrow as it is today. A model in the same workspace can call it without remembering the prompt.
2. **It has a stable input contract.** `client, invoice_number, line_items, tax_rate` — that's a schema. Schemas can be validated, serialized, sent over the wire.
3. **It produces a stable output.** A PDF in `/mnt/user-data/outputs/`, with a known shape. That output can be copied, mailed, stored, attached to a record.

Once you have those three things, the question stops being "how do I prompt the model better?" and becomes "what should call this skill, and when?"

## n8n: skills as workflow nodes

[n8n](https://n8n.io) is a node-based workflow engine — drag-and-drop automations that connect SaaS APIs, databases, and arbitrary HTTP endpoints. The Yambr platform exposes its tooling through the [LiteLLM](/integrations/litellm) gateway and the [hosted chat](/platform/hosted-chat), which means n8n can drive the same skills you've been calling from the chat UI.

The mental shift is small but important:

- In the chat, **a person triggers** a skill (by writing a prompt).
- In n8n, **an event triggers** the same skill (a webhook, a schedule, a row appearing in a database).

A concrete example: imagine your CRM emits an `invoice.requested` event with the deal data already structured. A two-node n8n workflow — *"on event → call invoice-builder skill → email the PDF to the customer"* — turns the model into a service. There is no chat, no human in the loop, and no prompt to maintain. The skill is just *there*, doing its job.

We are not going to walk through wiring this up in this course — that deserves its own dedicated page. The takeaway is: **the skill you built in scenario 8 is already n8n-ready**. The hard part was the packaging, not the integration.

When you're ready, see [Integrations → n8n](/integrations/n8n).

## What else falls out of having skills

- **CI/CD pipelines.** A skill that "generates the weekly customer-facing release notes from a Git diff" can run as a GitHub Action.
- **Scheduled reports.** A skill that "pulls metrics from BigQuery and emails a PDF every Monday at 8am" replaces a half-broken cron job.
- **Internal tools.** A skill that "takes a customer email and drafts a response based on the knowledge base" can be exposed as a Slack slash command via [LiteLLM](/integrations/litellm).
- **Multi-step agents.** A skill is itself a callable, so a higher-level [sub-agent](/features/sub-agents) can compose multiple skills (`scrape → summarize → email`) into a single behaviour.

The common thread is the same in every case. The skill is a **named function with a schema**. Once it exists, anything that can invoke a function can use it.

## A recap of the arc

The course had a deliberate structure. To make it explicit one more time:

| Stage | What you're doing | Cost per task | Repeatability |
|-------|-------------------|---------------|---------------|
| **Crawl** | Writing one-off prompts in the chat | High (tokens, time, attention) | Low (you copy-paste) |
| **Walk** | Using sub-agents and the live browser for richer tasks | Same as crawl | Same as crawl |
| **Run** | Calling skills you've built, by name | Low (just parameters) | High (deterministic) |
| **Automate** | Skills triggered by events, schedules, webhooks | Approaching zero | Total |

Each row is a step in leverage. The first three are inside the platform; the fourth is what happens when you connect the platform to the rest of your stack.

## What you should do next

If you've followed along and want to actually try this:

1. **Read the [Skills overview](/skills/overview)** to understand the on-disk format.
2. **Pick a recurring task** you do in the chat. Three repetitions is plenty of signal.
3. **Run `/skill-creator`** the way scenario 8 did. You don't need to write the SKILL.md by hand.
4. **Test the new skill in the chat** for a week. Refine the input contract until it feels obvious.
5. **Wire it into [n8n](/integrations/n8n)** — or whatever your team already uses for automations.

That's the loop. Skills are just prompts that grew up.
